Formats

Format Description Use case Pros Cons
.jpg, .jpeg Commonly used web image format Photographs, images that won’t suffer from compression
  • Small file size
  • Work on almost any device
  • Can’t display transparency
  • Lossy compression
  • Reds often have artifacts
  • Text can be hard to read when compressed
.png A lossless compression image format that allows for gradated transparency
  • Grayscale PNG: limited to black, white, and shades of gray
  • Indexed color PNG (PNG-8): limited to 256 colors
  • Truecolor PNG (PNG-24): can contain up to 16 million colors
 Limited color palettes, sharp lines, lots of red, and/or semi-transparency
  • Can display transparency at any value
  • Red will not show artifacts like in compressed JPEGs
  • Larger file size than JPEG format for the same picture
  • Most use cases are better served nowadays by SVG
.gif An older image format for static and animated images Limited color palettes and hard edges, binary transparency, animations
  • Allows for simple animations
  • Good when SVG isn’t supported
  • Easy to download, edit, and re-share
  • Very large file size, can slow down a page dramatically
  • Limited color palette
  • Does not work in Microsoft Outlook email client; will only display the first frame
  • Slow to render
.webp A newer image format that provides superior lossless and lossy compression Faster load times than JPEGs and PNGs in supported browsers
  • 26% smaller than PNGs, 25-34% smaller than JPEGs
  • Supports transparency
  • As of December 2021, still not natively supported in Adobe Photoshop
.svg Vector-based image format Low detail vector images, logos, icons; any graphics that need to scale without losing resolution or quality
  • Low file size when compared to raster images of the same size
  • Can change color with CSS
  • Can’t display photographs

Lazy loading

In pages with a lot of images, or on slow network connections, images can load very slowly. This has a fairly obvious negative impact on user experience, especially as screen resolutions increase and designers work with larger and larger images.

One way to improve page load speeds is to make graphics into vector files wherever possible. Logos and icons should be svg files, since those load orders of magnitude faster than the equivalent pixel-based files.

Another way is to add the loading="lazy" attribute to images. This tells browsers that support it to defer loading of images and iframes that are off-screen until the user scrolls near them.

All major browsers currently support this.

There are other ways to implement lazy loading via Javascript libraries, but now that it’s natively supported by mainstream browsers, it’s best practice to use the browser’s built-in implementation.

Responsive Images

Responsive images are a subset of image formats and usages that solve all of these four use cases:

  • Art direction: requesting visually distinct source files at specific screen sizes
    • Provide alternate versions of the same image for different screen sizes, whether it’s pixel dimensions or crop
    • Make sure all alternates can be described with the same alt attribute
  • Image types: requesting new image formats with responsible fallbacks
    • Provide fallbacks for browsers and devices that don’t support specific formats
    • Example: WEBP and SVG fallback to JPEG and PNG, respectively
    • Use the type="image/format" attribute to allow browsers to pick which format is best suited to the request
    • Limit wasted bandwidth by only serving one image for each use depending on what the browser supports
  • Display density: requesting image sources appropriate for high-density displays only when necessary
    • Only serve high-density images on high-density displays
    • If a display has 2x the pixel density, 300x300 images will actually have a 600x600 resolution
    • Have both 1x and 2x versions of an image so that browsers only load the one they need
  • Flexible images: providing the browser with information about a set of image sources and how they’ll be used for layout, so that it can choose the best image for a user's request
    • Hands control over to the browser; the designer doesn’t have explicit control over what image will display when
    • If a device’s pixel density is between two breakpoints, the browser will err on the side of the higher-resolution image
    • <img srcset="small.jpg 1.25x, medium.jpg 2.5x, large.jpg 5x" src="fallback.jpg" alt="...">
