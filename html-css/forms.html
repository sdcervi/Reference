Form basics

Forms can be used for anything from a simple search box to complex, multi-section application forms. Set them up with accessibility in mind from the beginning to make them work for as many users as possible.

Forms must be nested inside a form element:

<form action="http://www.example.com/form.php" method="GET" id="formId">
     ...
</form>

Attribute Values
action The URL for the page on the server that form data is submitted to
method GET: The values of the form are added to the end of the URL specified in the action attribute. Best for short forms or simple data retrieval. Not good for database actions.
POST: The values of the form are sent in HTTP headers. Best for uploaded files, very long forms, secure data such as passwords, and database actions.
id The HTML id of the form, usually used for CSS styling or JavaScript functions such as validation
autocomplete Tells the browser whether to allow autocomplete on form fields; this is generally defaulted to off for suspected login forms

Labels and groups

Use labels and groups to organize a form and make it easier to use.

Element HTML code HTML appearance Description & usage
Label <label for="inputName">Label text: </label>
<input type="text" name="inputName" id="inputName" />

Use labels to make forms easier to navigate, and more accessible to vision-impaired users. They can either wrap around the input they’re labeling (no for attribute needed in this context), or precede the input (for attribute needed). When setting a for attribute, use the id of the matching input or radio button.

When placing labels, put them above or to the left of text inputs, text areas, select boxes, and file uploads. Put them to the right of individual checkboxes or radio buttons.

Click on a field’s label to automatically select that field.
Fieldset with legend <fieldset>
     <legend>Legend text</legend>
     ...
</fieldset>
Legend text
Group related form controls together; this is helpful for longer forms with a lot of fields. Some browsers may render a border around fieldsets. Legends become headers for the fieldset.

Controls

Each form conrol must have a unique name for the server to know how to process all the data received. The name-value pairs sent to the server must match the code on the server, otherwise the form will not function correctly.

For newer HMTL5-specific form elements, older browsers will fallback to displaying it as a simple text input.

75		 This displays a numerical value depending on the numbers specified as attributes: low (min–low), medium (low—high), or high (high–max). Data validation

Client-side validation

HTML5 form elements (email, date, etc) have built-in validation in browsers that support them. However, this is not a replacement for server-side validation. Make sure to explicity define patterns and customize error messages for client-side validation. This will ensure better error handling and less load on the server for validation.

The CSS pseudo-class :invalid allows for styling of form fields that do not pass client-side validation.

Attribute Usage
pattern pattern attributes follow regular expressions. For example, to define a four-digit number, use [0-9]{4}. For accessibility, when a control has a pattern attribute, the title must describe the pattern.
required This specifies whether a form field needs to have data (of any type and format) in it before the form can be submitted.
minlength and maxlength These specify the minimum and maximum character counts of text-based data input.
min and max These specify the minimum and maximum value of numerical data input.
type This specifies whether the data input needs to be a number, email address, or other specific preset type. This can save time with certain data types to avoid having to set them with pattern, since the presets already have regular expressions specified.
