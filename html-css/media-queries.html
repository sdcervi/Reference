Media Queries

Media queries allow you to target specific screen sizes, ranges of sizes, or properties such as landscape or portrait orientation.

To add logic and conditionals to a media query, use the operators and, not, and only.

  • and: Combines multiple media features together into a single media query; each must evaluate to true for the media query to apply. This is also used to join media features with media types.
  • not: Negates a media query, returning true if the query evaluates to false; this applies to the entire query and not to individual features within a query. This requires a media type to be defined as part of the query.
  • only: Applies a CSS definition set only if every query matches. This helps prevent older browsers from incorrectly applying styles (Example: @media screen and (max-width: 500px) {...} would be applied to all screens). This requires a media type to be defined as part of the query.

Example: @media only screen and (min-width: 768px;) and (orientation: portrait) {...}

To add multiple queries to a single CSS definition set, separate them with a comma. Example: @media screen, print. This will mean that if any of the media queries evaluates to true, the CSS will apply.

Media types

Media types target a specific category of experience: all, print, or screen. The type is optional (default is all), except when using not or only.

Media type Usage
@media all All formats
@media only print Only print formats
@media not print Everything except print formats
@media only screen Only digital (screen) formats
@media not screen Everything except digital formats
@media screen, print Both digital and print formats

Media features

Media features describe specific characteristics of the user agent (browser), device, or environment the web page is being viewed in. These expressions test for their presence (as a boolean) or their value, and are optional. Each media feature expression needs to be in parenthesis.

Example: @media (max-width: 768px) {...}

If you create a media query without specifying a value (@media (color) {...}), the CSS rules will apply as long as the feature’s value is not 0 or none.

The table below only contains commonly used media features; here’s a full list of them.

Term Value Usage Example
aspect‑ratio [width]/[height] Width-to-height aspect ratio of the viewport (window), separated by a / @media (aspect‑ratio: 16/9) {...}
color number Number of bits per color component of the device, or 0 if the device is grayscale-only @media (color: 0) {...}
color‑gamut srgb, p3, or rec2020 The range of colors supported by the user agent and output device @media (color‑gamut: srgb) {...}
display‑mode fullscreen, standalone, minimal-ui, or browser Determines how a web app should run, depending on whether it’s running in-browser, as a standalone app, or in full screen. @media (display‑mode standalone) {...}
forced‑colors none or active Whether the browser should display colors normally or is using a restricted color palette such as High Contrast Mode. More information about what CSS this affects @media (forced‑colors: active) {...}
grid boolean (0 or 1) Whether the device uses a grid-based screen; almost all modern computers and mobile devices use bitmap screens. Grid-based screens include text-only terminals and basic phones with only one fixed font (i.e. digital dot matrix displays). @media (grid: 1) {...}
height [height]px The height of the viewport in pixels. You can also use min-height, max-height, or both to target ranges of screen sizes. @media (min‑height: 360px) {...}
orientation portrait or landscape Determines whether the device is in portrait orientation (height greater than or equal to width) or landscape orientation (width greater than height). This is especially useful for mobile devices, if you want to adapt the experience when a user rotates their device. @media (orientation: portrait) {...}
prefers‑color‑scheme light or dark Whether the user has light more or dark mode active; this can be either through the browser or their OS. This would allow for use of a different color palette to work better with dark mode or light mode backgrounds. @media (prefers‑color‑scheme: dark) {...}
prefers‑reduced‑motion no‑preference or reduce Whether the user has enabled the reduced motion feature on their browser and/or OS. This is common for users with vestibular motion disorders. The use case for this feature is to reduce or remove any interface animations. @media () {prefers‑reduced‑motion reduce}
resolution [resolution]dpi The pixel density of the device. You can also use min-resolution, max-resolution, or both to target ranges of resolutions. One use case for this is serving higher resolution images only on higher resolution displays. @media () {...}
scripting none, initial-only, or enabled Tests whether JavaScript is available. initial-only indicates that JavaScript is only supported on page load and not afterwards. @media (scripting: enabled) {...}
update none, slow, or fast How frequently (if at all) the device is able to modify (re-render) content. none means that a page is static after load and can’t be updated. slow means the refresh rate is slow and animations may not display quickly enough to appear smooth, like with e-readers or very slow devices. fast means the layout can change and re-render as needed and should appear smooth; most modern devices fall in this category. @media (update: slow) {...}
width [width]px The width of the viewport in pixels. You can also use min-width, max-width, or both to target ranges of screen sizes. @media (min‑width: 360px) {...}

Breakpoints

Breakpoints are where your page’s layout naturally breaks or looks strange. These become the points where you need to rearrange or restyle your layout to work better on the new screen size.

Best practices for design mean building websites from a mobile-first perspective: design it on the smallest screen size you want it to work on, then restyle as needed moving upward to each breakpoint. Frameworks like Bootstrap make this more straightforward for developers and designers, since their base styling system is designed for screens under 576px and scales up from there.

Designing mobile-first and then checking for breakpoints helps ensure a smooth, responsive, and consistent user experience across as many devices as possible. Rather than design for individual devices (iPhone, iPhone Plus, etc.), design for ranges of screen sizes to accomodate diverse users.
