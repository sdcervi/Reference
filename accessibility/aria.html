WAI-ARIA

The Web Accessibility Initiative (WAI) develops and maintains the Accessible Rich Internet Applications (ARIA) standards. These standards provide a clearly-defined framework for making websites, applications, and digital interfaces accessible to people who use technology in different ways. For example, it contains standards for how to make a webpage accessible and navigable via assistive technology (screen reader, keyboard navigation, etc).

When implementing ARIA in a webpage, this usually happens via roles and attributes. These supplement existing HTML elements when there isn’t a built-in mechanism for an accessibility need. For example, this makes forms, error messages, and JavaScript widgets accessible to disabled and impaired users. Whenever there is a native, built-in HTML element or attribute, use that, but for everything else, ARIA picks up the slack.

ARIA attributes can also be dynamically updated via JavaScript to reflect updated values in an element, using the same syntax as setting any other HTML attribute.

Remember: when implementing ARIA, do it correctly! No ARIA is better than incorrect or poorly-developed ARIA.

ARIA roles

ARIA roles provide semantic meaning to content, allowing screen readers and other tools to present and support interaction with object in a way that is consistent with user expectations of that type of object. ARIA roles can be used to describe elements that don’t natively exist in HTML or exist but don’t yet have full browser support.

Add an ARIA role to an element using the role attribute, set to one of the defined options (role="button"). Some roles require specific ARIA states to be included as well, or only work in association with other ARIA roles. A screen reader will announce elements based on their ARIA role if one is defined; make sure it accurately explains the semantic function of that element. Inaccurate ARIA roles negatively impact accessibility.

ARIA roles include six different categories:

  1. Document structure roles
  2. Widget roles
  3. Landmark roles
  4. Live region roles
  5. Window roles
  6. Abstract roles

Document structure roles

These provide a structural description for a section of content. Most of the defined roles in this category are no longer in use now that browsers now support semantic HTML elements that fill the same purpose.

Role Usage MDN documentation
toolbar A collection of common-used function buttons or controls. Only use this to group 3 or more controls. If there is more than one toolbar in a section, give each of them a unique aria-label or aria-labelledby to differentiate them. ARIA Roles: toolbar
tooltip A contextual text bubble that displays a description for an element, and appears when the cursor hovers over the element, or it has keyboard focus. Usually appears after a short delay. These are generally used for icons or terms. ARIA Roles: tooltip
feed A dynamic, scrollable list of articles in which articles are added or removed from either end as the user scrolls. ARIA Roles: feed
math Indicates that the content represents a mathematical expression. Content with this role is intended to be marked up with an accessible format such as MathML. If you have to use an image of the expression for browser compatibility, use the math role and make sure the alt attribute is correct and accurate. ARIA Roles: math
presentation/none Removes an element’s (and usually child elements’) semantic roles from being used for accessibility, while retaining the content. This more-or-less renders any semantic element as a plain div or span as far as screen readers and other accessibility devices are concerned. ARIA Roles: presentation
note Indicates a section whose content is parenthetical or ancillary to the main content of the page. Only use this if another native element such as aside doesn’t work. ARIA Roles: note

Widget roles

Widget roles are used to define common interaction patterns. Compared to document structure roles, widget roles usually require some JavaScript to make them fully interactive.

Role Usage MDN documentation
scrollbar Indicates a graphical object that controls the scrolling of content within a viewing area. ARIA Roles: scrollbar
searchbox Indicates an element is a type of textbox specifically for entering search criteria. ARIA Roles: searchbox
separator Indicates that the element is a divider used to visually separate and distinguish sections of content or items from each other. The hr element has an implicit role of separator. ARIA Roles: separator
slider Indicates an input element where the user selects a value from within a given range, using a visual slider. This is not to be confused with progress or meter, both of which are read-only. ARIA Roles: slider
switch Functionally identical to checkboxes, except the two states are on and off rather than checked and unchecked. ARIA Roles: switch
combobox An input that controls another element, such as a listbox or grid, that can dynamically pop up to help the user set the value of the input. ARIA Roles: combobox
tab An interactive element inside a tablist that, when activated, displays its associated tabpanel. ARIA Roles: tab
tabpanel The container for the instance of layered content associated with a tab. ARIA Roles: tabpanel
tablist The element that serves as the container for a set of tabs. ARIA Roles: tablist
treeitem An item in a hierarchical tree. ARIA Roles: treeitem
tree A widget that allows the user to select one or more treeitems from a hierarchically organized collection. ARIA Roles: tree
treegrid An element which is a grid whose rows can be expanded and collapsed in the same manner as a tree. ARIA Roles: treegrid

Landmark roles

These provide a way to identify the organization and structure of a webpage, when other implicit or explicit roles don’t fill in all the gaps. Use these sparingly; too many landmark roles can clog up a screen reader and make it more difficult to understand the structure of the page.

Role Usage MDN documentation
banner Defines a global site header that repeats sitewide. Usually includes a logo, company name, search bar, and possibly the navigation and/or slogan. The header element, when used for the site’s global header, has this role implicitly. ARIA Roles: banner
complementary Designates a supporting section that relates to the page’s primary content, but can stand on its own when separated. These are usually sidebars or call-out boxes. When possible, use the aside element instead. ARIA Roles: complementary
contentinfo Defines a global site footer that repeats sidewide. Usually include copyright information, navigation links, privacy policy and terms of use links, etc. Use the footer element instead when possible for semantic clarity. ARIA Roles: contentinfo
form Identifies a group of elements that function as a form. Use the form element instead when possible for semantic clarity. ARIA Roles: form
main Indicates the primary content of the page. Only use this once per page. ARIA Roles: main
navigation Used to identify major groups of links for navigating a site or section of content. Use the nav element instead when possible for semantic clarity. ARIA Roles: navigation
region Generic role for areas deemed significant but other landmark roles do not apply. Use the section element instead when possible for semantic clarity. ARIA Roles: region
search Identifies the section of the page used to conduct a search of any variety (sitewide, pagewide, or a collection of sites). Use this on the parent element containing the entire search module related to an input with type="search". When a form is solely dedicated to search, use the search role instead of the default form role. ARIA Roles: search

Live region roles

These define elements with content that can be dynamically changed. Using one of these roles helps screen readers identify when visual content has changed, so they can alert the user.

Role Usage MDN documentation
alert Indicates important, time-sensitive information. Only use this for elements that requre the user’s immediate attention, such as invalid form values, login session timeout warnings, etc. Add it to the element containing the alert message, not the element triggering the alert. This is considered an intrusive role, so use it sparingly and only where necessary. ARIA Roles: alert
log Identifies a region where new information is added in a meaningful order, and old information may disappear. This includes chat logs, message history, and error logs. New information is only added to the end, and existing information is ordered sequentially. ARIA Roles: log
marquee Contains non-essential information that changes frequently. For example: stock tickers, ad banners, latest posts in a topic or channel. The main difference between marquee and log is that log is presented sequentially. Make sure that marquees have an accessible label, either with aria-label or aria-labelledby. ARIA Roles: marquee
status Defines a region containing new information for the user that isn’t important enough to justify an alert. This is usually presented as a status bar. ARIA Roles: status
timer Indicates that an element is a numerical counter, counting up or down. Its contents should continuously update until the timer is paused or reaches its endpoint. ARIA Roles: timer

Window roles

These define sub-windows within the main window, such as modals and other pop-ups.

Role Usage MDN documentation
dialog Used to mark up an HTML-based application dialog or window that separates content from the rest of the page, such as a modal. These are generally placed on top of the rest of the page content using an overlay. dialogs can be either non-modal (can still interact with page content) or modal (only the content in the dialog can be interacted with). ARIA Roles: dialog
alertdialog Used on modal alert dialogs that interrupt a user’s workflow to communicate an important message and require a response. Examples include error messages that require confirmation, or other confirmation prompts such as permanently deleting information. ARIA Roles: alertdialog

Abstract roles

Abstract roles are only intended for browsers to use behind the scenes to organize and streamline documents. They should not be used by developers writing HTML markup.

ARIA attributes

ARIA attributes allow you to set specific properties and states to improve and streamline accessibility

Attribute Usage MDN documentation
aria-activedescendant Identifies the currently active element when focus is on a composite widget, combobox, textbox, group, or application. Set its value to the id of the currently focused element. ARIA Attributes: aria-activedescendant
aria-atomic Indicates whether assistive technologies like a screen reader present all or only parts of a changed live region, based on the change notifications defined with aria-relevant. Value is boolean. ARIA Attributes: aria-atomic
aria-autocomplete Indicates whether inputting text could trigger an autocomplete display. It can have two values: inline (displays a single predicted value, in place), or list (displays a list of possible values as a pop-out or drop-down). ARIA Attributes: aria-autocomplete
aria-braillelabel Defines a string value that labels the current element, with the intent to convert that value into Braille. ARIA Attributes: aria-braillelabel
aria-brailleroledescription Defines a human-readable, author-localized abbreviated description for the role of an element, with the intent to convert that value into Braille. ARIA Attributes: aria-brailleroledescription
aria-busy Indicates an element is being modified, and assistive technologies should wait until changes are complete before informing the user about the update. Value is boolean. ARIA Attributes: aria-busy
aria-checked Indicates the current checked state of checkboxes, radio buttons, and other widgets. Value is true, false, or mixed (as if in a folder tree where only some child folders are checked). When possible, use an input element with type="checkbox" as this has built-in semantics and does not require ARIA attributes. ARIA Attributes: aria-checked
aria-colcount Defines the total number of colums in a table, grid, or treegrid when not all columns are present in the DOM. Value is an integer greater than or equal to 1. ARIA Attributes: aria-colcount
aria-colindex Defines an element’s column index or position with respect to the total number of columns within a table, grid, or treegrid. Value is an integer greater than or equal to 1. ARIA Attributes: aria-colindex
aria-colindextext Defines a human-readable alternative to aria-colindex. For example, “Value at start of 2022.” ARIA Attributes: aria-colindextext
aria-colspan Defines the number of columns spanned by a cell or gridcell within a table, grid, or treegrid. Value is an integer greater than or equal to 1. ARIA Attributes: aria-colspan
aria-controls Identifies the element or elements whose contents or presence are controlled by the element that has this attribute. Set its value to the id or space-separated ids of the controlled element. ARIA Attributes: aria-controls
aria-current A non-null value indicates that this element represents the currently focused item within a container or set of related items. Values can include page, step, location, date, time, true, or false. Any other non-null string value will be treated as true. If the attribute is missing, is an empty string, has no value assigned, or is set to false then it is not exposed to assistive technology. When something is selected rather than current, i.e. a tab in a tablist, use aria-selected instead. ARIA Attributes: aria-current
aria-describedby Identifies the element that describes the element on which this attribute is set. The value is a single or space-separated list of ids of elements that describe the current one. This is similar to aria-labelledby; aria-labelledby lists the elements describing the essence of an object, where aria-describedby lists the elements providing more information that the user might need (a short caption/heading rather than a longer description). The aria-describedby target content should be a plain text string. For more complex descriptions, use aria-details. ARIA Attributes: aria-describedby
aria-description Contains a string value that describes or annotates the current element. This is similar to aria-label; aria-label should be short and concise, while aria-description can have more detail. Use this when no descriptive text is availabe in the DOM; use aria-described-by when descriptive text is already present in the DOM. ARIA Attributes: aria-description
aria-details Identifies the element that describes the element on which this attribute is set. The value is a single or space-separated list of ids of elements that describe the current one. This is similar to aria-labelledby; aria-labelledby lists the elements describing the essence of an object, where aria-details lists the elements providing additional information or complex descriptions. For simpler plain-text descriptions, use aria-describedby. ARIA Attributes: aria-details
aria-disabled Indicates the element is perceivable but disabled, for example a disabled button. Value is boolean, and a missing or null value is considered false. ARIA Attributes: aria-disabled
